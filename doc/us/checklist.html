Release Checklist

This document has been adapted from the Module Release Checklist written by Andre Carregal for the Kepler Project

This checklist describes the release process of LuaEclipse

Once a release cycle is announced in the LuaEclipse Developers list, most of the tasks are supposed to be executed by developers with CVS commit access. But if you want to help with the checklist, tasks that can be executed by anyone with CVS read access are marked with a "(anyone)" tag. Just make sure you use the module CVS HEAD and report any issues in the LuaEclipse Developers list.

  1. Check merges on the module release branch and the trunk (project developers)
  2. Check pending bug reports and feature requests (anyone)
  3. Prepare distribution
    1. Publish the Preview Documentation (project developers)
      • Check previous version status
      • transfer contents to luaforge ftp site
    2. Publish the Preview Update-site (project developers)
      • Check previous version status
      • transfer contents to luaforge ftp site
    3. Announce the Preview version (project developers)
      • Announce on developers list
      • Establish date for full release
    4. Check plug-in dependencies (anyone)
    5. Check LuaJava (anyone)
      1. LuaJava version (see that we are using the latest version)
      2. LuaJava binaries (see that shared libraries are present - and actually work - for all supported platforms)
    6. Check the Java source of all plug-ins (anyone)
      1. Header comments (see that copyright, permission and disclaimers are in all sourcefiles)
      2. JavaDoc comments (see that all Java classes have JavaDoc comments)
      3. CVS tagline (see that all Java classes have the $Id$ CVS tag present as @version)
    7. Check plug-in settings (anyone)
      1. Plug-in manifest files (see that the plug-in version, name and vendor are correct)
      2. Plug-in dependencies (see that all plug-in dependencies are met, check runtime classpaths and version numbers)
      3. Plug-in build properties (see that all files needed by the plug-in are included in build)
    8. Check feature and update-site settings (anyone)
      1. Feature manifest (see that feature version, name and vendor are correct; check license, copyright and description)
      2. Update site information (see that update site name and URL are correct on both projects: feature and update-site)
      3. Feature dependencies (see that all plug-ins are present on feature, with correct versions )
    9. Check the documentation (anyone)
      1. Version
      2. License
      3. What's new
        • Check with CVS history if nothing has been forgotten. You can get a full listing of deltas for the working version using cvs2cl -t.
      4. Release date
      5. References to [=LuaJava=] and its version
      6. Images (PNG, GIF)
      7. Lua compatibility
      8. Eclipse compatibility
      9. Platform compatibility
      10. Dependencies on other plug-ins
      11. Applicability of screenshots
      12. Documentation Links
        • Links to previous versions(?).
        • Download from LuaForge(http://luaforge.net/projects/luaeclipse/files)
        • Links to LuaEclipse Home (including the logos)
        • Links to the mailing list on the Contact Us section (on index.html) and on the left menu
        • Links to the Lua Site
        • Links to the license page
        • Link to the project LuaForge sections on the left menu:
          • Project home
          • Bug Tracker
          • CVS
      13. Validate XHTML and links. This can be done locally or remotely. If done remotely the files should be copied to a ''preview'' directory on (http://www.keplerproject.org/projectname/preview). A simple link checker can be found at http://validator.w3.org/checklink
      14. README (?)
        • Reference to the project home page
        • Version number
        • Description of the directories and files
      15. Test the examples in the documentation
    10. Execute the unit tests (anyone)
  4. CVS details (project developers)
    • Branch using rel_[0-9]_[0-9]_x
    • Tag using v[0-9]_[0-9]_[0-9]_[alpha|beta][2-9]
  5. Generate the tar.gz and zip archives (using the Distribution Format) (project developers)
    • Standard distribution format luaeclipse-X.Y[.Z].zip
    • Source files tarball luaeclipse-src-X.Y[.Z].tar.gz
  6. Test the tgz and zip archives. (anyone)
    • Check included files
    • Check the install/build process.
  7. LuaForge project details (project admin)
    • Bug Tracker
      • Categories (API, Documentation, Other)
      • Groups (Version numbers)
    • Feature Requests
      • Categories (API, Documentation, Other)
      • Groups ("Next Release")
  8. Upload to LuaForge (project developers)
    • Create the release LuaEclipse-X.Y[.Z]
    • Release and Change notes (the line breaks have to be manually positioned)
    • Source tar.gz files
    • Binary zip files
  9. Upload to LuaForge ftp site (ftp.luaforge.net) (project developers)
    • Publish the Final Documentation (project developers)
      • Documentation
      • Logo and images
    • Publish the Final Update-site (project developers)
  10. Test the Final Update-site. (anyone)
    • Test on supported platforms
    • Test on supported Eclipse versions
  11. Update LuaForge project page (project admin)
    • Public Info
    • Trove version and scope
  12. Announce the Release on the Developers list (project admin)
  13. Announce the Release on the Kepler list (project admin)
  14. Announce on the Release on LuaForge as a news post (project admin)
    • LuaForge Home Approval (LuaForge admin)
  15. Announce the Release on the Lua list (project admin)
